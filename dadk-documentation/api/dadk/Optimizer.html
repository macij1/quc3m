Optimizer(Tab)#

class dadk.Optimizer.Optimizer(**kwargs: Any)#

Main GUI-class of DADK’s prototyping framework. The constructor takes a OptimizerModelComplex model and creates a GUI within a Jupyter notebook. The latter consists of tabs for the different methods of the model. In particular, tabs can be used to change parameters of different model methods, trigger the methods via a Call button, reset settings to the parameters as stored in the dao-file via a Reset button, change to default settings via a Default button and save settings in the dao-file via a Save button. An Optimizer object can also be constructed and run in batch_mode, i.e. without creating a GUI. In this case either the immediate_run or the immediate_load parameter should be set to True such that the run through the different model methods is started immediately upon construction.

Parameters:

  • model (OptimizerModelComplex) – Model for this Optimizer

  • mode (str) – Special mode for filtering widgets. Among widgets in the model which were declared with a list of tags, only those which have mode in their list of tags are shown

  • auto_load (bool) – show progress bar for autoload

  • immediate_load (bool) – Immediately load the model with parameters and solutions read from dao-file

  • immediate_run (bool) – Immediately load the model with parameters read from dao-file

  • show_load (bool) – Show or hide Setup scenario tab

  • show_compose (bool) – Show or hide Compose model tab

  • show_build (bool) – Show or hide Build QUBO tab

  • show_solve (bool) – Show or hide Solve annealing tab

  • show_track (bool) – Show or hide Anneal tracker tab

  • show_report (bool) – Show or hide Report tabs

  • show_visual (bool) – Show or hide Visualization tab

  • show_test (bool) – Show or hide Pareto tab (old name)

  • show_pareto (bool) – Show or hide Pareto tab

  • show_timeseries (bool) – Show or hide Timeseries tab

  • show_optuna (bool) – Show or hide Optuna tab

  • read_only (bool) – If True, the underlying dao-file is not modified

  • batch_mode (bool) – True if running from the Optimizer-GUI framework, else False

  • qubo_hash_step_size (int) – Defines granularity of md5sum check on QUBOs

  • stop_replay_on_error (bool) – If True and in case the replay cannot continue, an exception is raised. Otherwise, process is continued with an additional annealing call instead of the replay

  • log_level (int) – Define log-level, default is logging.ERROR

  • log_format (str) – Define log-formatting, default is ‘%(asctime)s - %(levelname)s - %(message)s’

  • overload_print (bool) – Overload print statement, only used for running compose-model with multi-threading

  • use_default_compose_model (bool) – If True, adds a simple compose_model method consisting of one call of build_and_solve, even when not declared

  • development_mode (bool) – Run optimizer in development mode. Addition “generate” - button is added to the tabs, to generate the required method body.

optuna#

Optimizer.optuna(regenerate: str = '', call_clear_output: bool = True)#

The optuna option is offered by optimizer, if a parameter optuna_setting different from None was defined. Then it is expected, that the used OptimizerModel defines a method optuna, which returns a float value. The method is called after prep_result and returns a quality metric for the test case. This value is used for a minimization study with optuna, typically the energy of the underlying QUBO HP optionally plus energy of penalty polynomial HP is returned.

The method generates a parameter tuning series by optuna and displays the results. The parameter regenerate allows 2 modes:

  • False: no generate is done, results from an earlier run are displayed

  • True: a complete new test series is executed and replaces the a possible series from earlier run

The parameters for the test series are derived from current setting of the Optimizer object as can be found in the associated persistent dao file. Certain parameters are varied in the specific tests according to the parameter optuna_setting in the model constructor. The parameter is expected as dictionary with arrays for the keys var_object, var_attribute and var_values. For example 'var_object': ['load_parameter', 'build_qubo_parameter', 'solver_parameter'] refers to three parameters to be propagated to the load method, the build_qubo method and the solve method. A corresponding array 'var_attribute': ['city_count, 'factor_A', 'temperature_start'] points to the parameter names used in the respective method while 'var_values': [[3, 5, 20], [1000, 1100, 1200], [100.0, 800.5, 700.0]] defines the values to be used in the test series. Beside these categorical definitions as lists of values fitting to the respective parameter types is is possible to define spans for parameters of type int or float. This is specified by tuples consisting of type, lower and upper bound; in the example above one could define 'var_values': [(int, 3, 20), (int, 1000, 1200), (float, 100.0, 700.0). Optuna will then execute a study within the cartesian product of categorical value lists and spans. With the optional key trials the maximum number of test can be limited, by default 200 test will be executed.

For every test case a numbered dao file 0.dao, 1.dao … is created inside the optuna directory, which has relative to the working directory the path <base_dao_file_name_without_suffix>/optuna>. A summary of the study can be found as pickle string under optuna_data in the main dao file. The varied parameters and results are stored in the base dao file as well. It is possible to copy the optimum settings directly to the dao file. The functions detect from the environment possibly changed parameters and displays a button for regenerate; this button calls the optuna method with appropriate arguments to generate from scratch optuna study.

Parameters:

  • regenerate (str) – string with the admissible values ‘’ (NO_GENERATE), ‘from scratch’ (FROM_SCRATCH) and ‘incremental’ (INCREMENTAL)

  • call_clear_output (bool) – default is True, this will clear the output for each nested Optimizer. If set to False, you will be able to see all nested Optimizer for debugging.

pareto#

Optimizer.pareto(regenerate: str = '', call_clear_output: bool = True)#

The pareto option is offered by optimizer, if a parameter pareto_setting different from None is defined. Then it is expected, that the used OptimizerModel defines a method pareto, which returns a dictionary. The method is called after prep_result and returns under the keys of the dictionary the values to be recorded for later test series examination; typically these are numeric values to allow visualization as graphs.

The method generates a test series and displays the results. The parameter regenerate allows 3 modes:

  • empty string (NO_GENERATE) or False: no generation of a new test series is done, instead results from an earlier run are displayed

  • ‘from scratch’ (FROM_SCRATCH): a complete new test series is executed

  • ‘incremental’ (INCREMENTAL): an earlier and incomplete test series is recovered and continued

The parameters for the test series are derived from current settings of the Optimizer object as can be found in the associated persistent dao file. Certain parameters are varied in the specific tests according to the parameter pareto_setting in the model constructor. The parameter is expected as dictionary with arrays for the keys var_object, var_attribute and var_values. For example 'var_object': ['load_parameter', 'build_qubo_parameter', 'solver_parameter'] refers to three parameters to be propagated to the load method, the build_qubo method and the solve method. A corresponding array e.g. 'var_attribute': ['city_count, 'factor_A', 'temperature_start'] points to the parameter names used in the respective method. Finally, e.g., 'var_values': [[3, 5, 20], [1000, 1100, 1200], [100, 800, 700]] defines the values to be used in the test series. For the test series, the values are varied simultaneously. The respective entry in each array with a certain index defines the test case of that index. The example above will produce 3 test cases. Using the optional key 'cartesian_product': True in the pareto setting dictionary will process the cartesian product of the value lists. In the example above this results in 27 tests.

For every test case a numbered dao file 0.dao, 1.dao … is created inside the pareto directory, which has relative to the working directory the path <base_dao_file_name_without_suffix>/pareto>. A summary of the measured results can be found as html and json in the files summary.html and summary.json. The varied parameters and results are stored in the base dao file as well. This is the source for visualization. The functions detect from the environment possibly changed parameters or incomplete earlier test series and displays buttons for recovery and regenerate; these buttons call the pareto method with appropriate arguments to generate from scratch or incremental.

Parameters:

  • regenerate (str) – string with the admissible values ‘’ (NO_GENERATE), ‘from scratch’ (FROM_SCRATCH) and ‘incremental’ (INCREMENTAL)

  • call_clear_output (bool) – default is True, this will clear the output for each nested Optimizer. If set to False, you will be able to see all nested Optimizer for debugging.

The class OptimizerModel structures the process for optimization in phases for data loading and preparation, model construction, solving the mathematical problem and preparing and translating the result into to business terms of the original problem. Objects of this class are used to implement a complete optimization chain. In particular, the user implements a loading method load to load data and setup the underlying business model, a build routine (build_qubo method) to generate the optimization model from the actual business model and a routine to process the optimization result (prep_result method). Also a compose_model routine can be used for iterative optimization algorithms. The user defines the control logic for the iterative process in the corresponding method.

An instance of a OptimizerModel objects can be used as a parameter to construct an Optimizer object, which then controls the processes for the given OptimizerModel. It generates a GUI within a Jupyter notebook which allows to handle the different routines of the model via a dashboard with tabs and widgets. Time information for composed and single optimization algorithms are automatically tracked. Routines for result presentation (report_<number> method) and visualization (draw method) can be executed if they were implemented for the OptimizerModel. If the model defines parameter spans and target functions then the Optimizer can execute pareto or optuna test series to achieve parameter optimization with respect to a user defined optimization target.

The file with name persistent_file stores all parameter values of the model as well as information on the latest execution of the models methods. In combination with an Optimizer object, the model’s persistent_file can be used to replay the optimization. The Excel file with name profile_file is by default used to store time information on the solution process if the model’s OptimizerModel.compose() method is called. The user can also choose to store plain text information or dictionary like information in this file by using the model’s methods OptimizerModel.log() and OptimizerModel.log_data().

The different steps of the optimization algorithm are seperated in methods of the model and presented in different tabs in the model’s Optimizer object. All actions for loading the model are typically implemented in the OptimizerModel.load() method. For parameters of this method which the user wants to make adjustable in the GUI of the Optimizer object, widgets can be declared as WidgetDefinition dictionaries in the list load_parameter. Each such dictionary must contain values for the keys name, type the initial value and all other information which are necessary to build the widget. The user defined value of each wigget will then given as parameter named as the corresponding value for the name key to the method OptimizerModelComplex.load() upon execution e.g. via the Call button of the Optimizer`s GUI.

Similarly, the build_qubo_parameter are used to create a GUI which return the actual values to the build_qubo() method. The Solve annealing parametrization depends on the used solver. This is completely specified for all supported solvers and needs no extra efforts. The only thing is the implemention of the method prep_results(), is called after successful optimization to re-translate the bit-strings of the solution back to business data representation. This can be used by the report() or draw() message, to visualize the results.

If a compose_model() method has been implemented, then the same procedure for parametrization takes place as for load(). For execution there are two options for compose_model(). If the Call button is pressed, the method is executed and the whole process is driven. The Start button will drive the process only to the first breakpoint, which is normally after the first call of build_and_solve or pass #breakpoint within the compose_model() method. The parameter compose_model_break_after_statement defines a regular expression to detect breakpoints in the code; it is possible to modify this behaviour, e.g. compose_model_break_after_statement = 'pass #breakpoint' would stop at pass #breakpoint but not at any build_and_solve.

OptimizerModel supports pareto and time series tests. Test settings are defined as dictionaries in the parameters pareto_setting and time_series_setting. To define the tests, the user specifies which parameters should be set in each test. The list under the key ‘var_object’ of the setting dictionary contains a list which refers to the names of the methods for which a parameter should be set: load_parameter, build_qubo_parameter, solve_parameter. For each entry, the list under the key ‘var_attribute’ must contain a name of a parameter of the corresponding method for which values for the tests will be defined. The actual list of values (one list for each attribute entry) for the tests are specified in a list under the keyword ‘var_values’. For pareto test all list under ‘var_values’ must have the same length and the values with the same index from all lists are combined to one test case. Setting the key/value pair cartesian_product': True changes the test case generation to the cartesian product of all lists; that typically increases the number of tests enormously, so handle with care. For timeseries always the cartesian product is generated. All information of the pareto test which should be stored for analysis have to be defined in the method OptimizerModel.pareto(); the return value of the method should be a dictionary with information specified as key-value pairs typically with numeric values. After running the pareto test the GUI offers many possibilities to visualize the recorded curves.

..pareto_setting example:

pareto_setting={
    'cartesian_product': False,
    'var_object':    ['load_parameter',      'build_qubo_parameter', 'solve_parameter'],
    'var_attribute': ['min_service',         'lambda_A',             'time_limit_sec'],
    'var_values':    [[40000, 70000, 90000], [500, 700, 900],        [10, 15, 25]]
},

Similarly, parameter optimization leveraging the hyperparameter optimization Optuna is supported. The parameters to be optimized are specified in a dictionary under the parameter optuna_setting. As for pareto and time series tests, the parameters to be optimized are defined with lists under the keys ‘var_object’, ‘var_attribute’ and ‘var_values’. Objects of this class can be used to implement a complete optimization chain. In particular, the user defines a loading routine load as a method to load data and define the underlying model, a build routine (build_qubo method) to generate the optimization task and a routine to process the optimization result (prep_result method). Also a compose_model routine can be used for iterative optimization algorithms or for algorithms which require several optimization processes. The user has to define the corresponding method. Time information for composed and single optimization algorithms are automatically tracked. Routines for result presentation (report_<number> method) and visualization (draw method) can be implemented in respective methods. The class also enables the execution of pareto and time series tests via the respective methods pareto and time_series. Parameter optimization with respect to a user defined optimization target can be defined in a corresponding method optuna.

OptimizerModelComplex objects can be used as a parameter to construct an Optimizer object. The latter generates a GUI within a Jupyter notebook where the different routines of the model can be handled via a dashboard with tabs and widgets.

OptimizerModelComplex also supports usage of multiple tags for build and solve tabs/routines.

The file with name persistent_file stores all parameter values of the model as well as information on the latest execution of the models methods. In combination with an Optimizer object, the model’s persistent_file can be used to replay the optimization. The Excel file with name profile_file is by default used to store time information on the solution process if the model’s OptimizerModelComplex.compose() method is called. The user can also choose to store plain text information or dictionary like information in this file by using the model’s methods OptimizerModelComplex.log() and OptimizerModelComplex.log_data(). The different steps of the optimization algorithm are seperated in methods of the model and presented in different tabs in the model’s Optimizer object. All actions for loading the model are typically implemented in the OptimizerModelComplex.load() method. For parameters of this method which the user wants to make adjustable in the GUI of the Optimizer object, widgets can be declared as dictionaries in the list load_parameter. Each such dictionary must contain values for the keys name, type the intitial value and all other information which are necessary to build the widget. The initial value will then given as parameter name to the method OptimizerModelComplex.load() upon execution e.g. via the Call button of the Optimizer`s GUI. Similarly, parameter definitions for the methods OptimizerModelComplex.compose_model() can be defined in the list compose_model_parameter. For step-wise execution within the Optimizer framework, a string load_break_after_statement can be used to interrupt execution of the OptimizerModelComplex.load() method whenever the corresponding statement appears in the method’s code. Execution can be continued by clicking a Step or a Run button. Note that load_break_after_statement must be a complete statement (line). The same functionality can be used in the OptimizerModelComplex.compose_model() method by using the parameters compose_model_break_after_statement. OptimizerModelComplex supports tagging ob build_qubo and prep_result methods. The corresponding templates are defined in the dictionaries build_qubo_templates and solve_templates. Each tag has a string as key and a template dictionary as value. For each tag in build_qubo_templates a corresponding tag in solve_templates must be defined. In the build template dictionary, the value under the key KEYWORD_ON_CALL specifies the function to be called upon execution. Typically, this is a method defined by the user. The value under the key KEYWORD_PARAMETER is a dictionary with widget definitions as for the OptimizerModelComplex.load() and OptimizerModelComplex.compose_model() method. Each of the widgets must have a name which is also a parameter of the function specified by KEYWORD_ON_CALL. In the solve template dictionary, default solver parameters can be defined in the dictionary under the key KEYWORD_DEFAULT_PARAMETER. Calculated solver parameters can be defined in the dictionary under the key KEYWORD_CALCULATED_PARAMETER. The function to be called upon execution of the template (typically a user defined method) is specified under the keyword KEYWORD_PREP_RESULT. Execution of the build and solve template can be triggered by the user in the GUI or by the model’s method OptimizerModelComplex.build_and_solve() with the tag specified as parameter tag. Toolbar parameters can be defined for the tabs compose_model, build_qubo, visualization, pareto, time_series and optuna. as respective parameters <tab>_toolbar_parameter. Each parameter contains a list of dictionary descriptions for widgets to be shown in the top of the tab. on_click and similar events can be defined to control the actions to be performed when executing the widgets changing the widget values.

OptimizerModelComplex supports pareto and time series tests. Test settings are defined as dictionaries in the parameters pareto_setting and time_series_setting. To define the tests, the user specifies which parameters should be set in each test. The list under the key ‘var_object’ of the setting dictionary contains a list which refers to the names of the methods for which a parameter should be set: load_parameter, build_qubo_parameter, solve_parameter. For each entry, the list under the key ‘var_attribute’ must contain a name of a parameter of the corresponding method for which values for the tests will be defined. The actual list of values (one list for each attribute entry) for the tests are specified in a list under the keyword ‘var_values’. For pareto test all list under ‘var_values’ must have the same length and the values with the same index from all lists are combined to one test case. Setting the key/value pair cartesian_product': True changes the test case generation to the cartesian product of all lists; that typically increases the number of tests enormously, so handle with care. For timeseries always the cartesian product is generated. All information of the pareto test which should be stored for analysis have to be defined in the method OptimizerModel.pareto(); the return value of the method should be a dictionary with information specified as key-value pairs typically with numeric values. After running the pareto test the GUI offers many possibilities to visualize the recorded curves.

pareto_setting={
    'cartesian_product': False,
    'var_object':    ['load_parameter',      'build_qubo_parameter', 'solve_parameter'],
    'var_attribute': ['min_service',         'lambda_A',             'time_limit_sec'],
    'var_values':    [[40000, 70000, 90000], [500, 700, 900],        [10, 15, 25]]
},

class dadk.Optimizer.InterruptKey(status_output: Output = None)#
