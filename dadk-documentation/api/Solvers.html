QUBOSolverDAv3c(RestMixin, AzureBlobMixin, QUBOSolverBase)#

class dadk.QUBOSolverDAv3c.QUBOSolverDAv3c(time_limit_sec: int, target_energy: int = None, num_group: int = 1, num_solution: int = 16, num_output_solution: int = 5, gs_num_iteration_factor: int = 5, gs_num_iteration_cl: int = 800, gs_ohs_xw1h_num_iteration_factor: int = 3, gs_ohs_xw1h_num_iteration_cl: int = 100, ohs_xw1h_internal_penalty: int | str = 0, gs_penalty_auto_mode: int = 1, gs_penalty_coef: int = 1, gs_penalty_inc_rate: int = 150, gs_max_penalty_coef: int = 0, scaling_action: ~dadk.internal.ScalingParameter.ScalingAction = ScalingAction.NOTHING, scaling_factor: int | float = 1.0, scaling_bit_precision: int = 64, guidance_config: ~dadk.BinPol.PartialConfig | None = None, connection_parameter: ~dadk.internal.ConnectionParameter.ConnectionParameter = None, offline_request_file: str = None, offline_response_file: str = None, random_seed: int = None, logger: ~logging.Logger = <Logger dadk.QUBOSolverDAv3c (ERROR)>)#

Objects of QUBOSolverDAv3c can be used to solve quadratic binary optimization problems (qubo) with quadratic binary constraints (penalty qubo) and (several) linear binary inequality constraints. In particular, the problem is solved with a Digital Annealer version 3c solver, a hybrid solver. The constructor takes solver parameters of the Digital Annealer version 3c solver as described in the API documentation , parameters for automatic scaling of the qubo and connection parameters to establish the connection to a Digital Annealer version 3c solver. The solver object can be used to solve (several) optimization problems with those fixed parameters via the minimize method.

Parameters:

  • time_limit_sec (int) – Maximum running time of DA in seconds. Specifies the upper limit of running time of DA. Time_limit_sec should be selected according to problem hardness and size (number of bits). GUI: General / Time limit [sec] , Min: 1, Max: 3600

  • target_energy (int) – Threshold energy for fast exit. This may not work correctly if the specified value is larger than its max value or lower than its min value. GUI: General / Target energy , Default: None, Min: -99999999999, Max: 99999999999

  • num_group (int) – Number of independent optimization processes. Increasing the number of independent optimization processes leads to better coverage of the search space. Note: Increasing this number requires to also increase time_limit_sec such that the search time for each process is sufficient. GUI: General / Number optimizations , Default: 1, Min: 1, Max: 16

  • num_solution (int) – Number of solutions maintained and updated by each optimization process. GUI: General / Solutions per optimization , Default: 16, Min: 1, Max: 1024

  • num_output_solution (int) – Maximal number of the best solutions returned by each optimization. Total number of results is num_solution * num_group. GUI: General / Results per optimization , Default: 5, Min: 1, Max: 1024

  • gs_num_iteration_factor (int) – Maximal number of iterations in one epoch of the global search in each optimization is gs_num_iteration_factor * number of bits. GUI: Global search / Iterations factor , Default: 5, Min: 0, Max: 100

  • gs_num_iteration_cl (int) – Maximal number of iterations without improvement in one epoch of the global search in each optimization before terminating and continuing with the next epoch. For problems with very deep local minima having a very low value is helpful. GUI: Global search / No improv. cutoff , Default: 800, Min: 0, Max: 1000000

  • gs_ohs_xw1h_num_iteration_factor (int) – Maximal number of iterations in one epoch of the global search in each optimization is gs_ohs_xw1h_num_iteration_factor * number of bits. Only used when 1Hot search is defined. GUI: Global search / 1Hot iterations factor , Default: 3, Min: 0, Max: 100

  • gs_ohs_xw1h_num_iteration_cl (int) – Maximal number of iterations without improvement in one epoch of the global search in each optimization before terminating and continuing with the next epoch. For problems with very deep local minima having a very low value is helpful. Only used when 1Hot search is defined. GUI: Global search / 1Hot no improv. cutoff , Default: 100, Min: 0, Max: 1000000

  • ohs_xw1h_internal_penalty (Union[int,str]) –

    Mode of 1hot penalty constraint generation.

    0:

    internal penalty generation off: 1hot constraint as part of penalty polynomial required

    1:

    internal penalty generation on: 1hot constraint not as part of penalty polynomial required

    If 1way 1hot constraint or a 2way 1hot constraint is specified, ohs_xw1h_internal_penalty = 1 is recommended. GUI: Penalty polynomial / 1Hot penalty mode , Default: 0, Min: 0, Max: 1

  • gs_penalty_auto_mode (int) –

    Parameter to choose whether to automatically incrementally adapt gs_penalty_coef to the optimal value.

    0:

    Use gs_penalty_coef as the fixed factor to weight the penalty polynomial during optimization.

    1:

    Start with gs_penalty_coef as weight factor for penalty polynomial and automatically and incrementally increase this factor during optimization by multiplying gs_penalty_inc_rate / 100 repeatedly until gs_max_penalty_coef is reached or the penalty energy is zero.

    GUI: Penalty polynomial / Penalty auto mode , Default: 1, Min: 0, Max: 1

  • gs_penalty_coef (int) – Factor to weight the penalty polynomial. If gs_penalty_auto_mode is 0, this value does not change. If gs_penalty_auto_mode is 1, this initial weight factor is repeatedly increased by gs_penalty_inc_rate until gs_max_penalty_coef is reached or the penalty energy is zero. GUI: Penalty polynomial / Penalty coefficient , Default: 1, Min: 1, Max: 9223372036854775807

  • gs_penalty_inc_rate (int) – Only used if gs_penalty_auto_mode is 1. In this case, the initial weight factor gs_penalty_coef for the penalty polynomial is repeatedly increased by multiplying gs_penalty_inc_rate / 100 until gs_max_penalty_coef is reached or the penalty energy is zero. GUI: Penalty polynomial / Penalty increase rate , Default: 150, Min: 100, Max: 200

  • gs_max_penalty_coef (int) – Maximal value for the penalty coefficient. If gs_penalty_auto_mode is 0, this is the maximal value for gs_penalty_coef. If gs_penalty_auto_mode is 1, this is the maximal value to which gs_penalty_coef can be increased during the automatic adjustment. If gs_max_penalty_coef is set to 0, then the maximal penalty coefficient is 2^63 - 1. GUI: Penalty polynomial / Max penalty coefficient , Default: 0, Min: 0, Max: 9223372036854775807

  • scaling_action (ScalingAction) –

    Following methods for scaling qubo, penalty_qubo and inequalities are available:

    ScalingAction.NOTHING:

    no action

    ScalingAction.SCALING:

    scaling_factor is multiplied to qubo, penalty_qubo and inequalities.

    ScalingAction.AUTO_SCALING:

    A maximum scaling factor w.r.t. scaling_bit_precision is multiplied to qubo, penalty_qubo and inequalities.

  • scaling_factor (Union[int, float]) – The scaling_factor for qubo, penalty_qubo and inequalities.

  • scaling_bit_precision (int) – Maximum scaling_bit_precision for qubo, penalty_qubo and inequalities. Used to define the scaling factor for qubo, penalty_qubo and inequalities.

  • guidance_config (Optional[PartialConfig]) – Specifies an initial value for each polynomial (problem) variable that is set to find an optimal solution. By specifying a value that is close to the optimal solution, improvement in the accuracy of the optimal solution can be expected. If you repeatedly use the specified initial values to solve the same polynomial (problem), the same optimal solution is obtained each time.

  • connection_parameter (ConnectionParameter) – The user defined dadk.internal.ConnectionParameter.ConnectionParameter object

  • offline_request_file (str) – Filename for storing of request JSON document.

  • offline_response_file (str) – Filename for storing of response JSON document.

  • random_seed (int) – Seed for random numbers for the optimization call, valid values are [0, 9999].

  • logger (Logger) – The logger used for this solver

class dadk.QUBOSolverCPU.QUBOSolverCPU(optimization_method: ~dadk.internal.DAv2Parameter.OptimizationMethod = OptimizationMethod.annealing, number_runs: int = 2, number_replicas: int = 5, number_iterations: int = 1000, temperature_sampling: bool = True, temperature_start: float = 1000.0, temperature_end: float = 1.0, temperature_mode: int = 0, temperature_interval: int = 1, offset_increase_rate: float = 5.0, pt_temperature_model: str = 'Exponential', pt_replica_exchange_model: str = 'Neighbours', solution_mode: ~dadk.internal.DAv2Parameter.SolutionMode = SolutionMode.COMPLETE, graphics: bool = False, scaling_action: ~dadk.internal.ScalingParameter.ScalingAction = ScalingAction.NOTHING, scaling_factor: int | float = 1.0, scaling_bit_precision: int = 64, guidance_config: ~dadk.BinPol.PartialConfig | None = None, connection_parameter: ~dadk.internal.ConnectionParameter.ConnectionParameter = None, offline_request_file: str = None, offline_response_file: str = None, random_seed: int = None, logger: ~logging.Logger = <Logger dadk.QUBOSolverCPU (ERROR)>)#

Solver implementation for DA version 2 on CPU.

The constructor of QUBOSolverCPU sets all parameters to define the annealing process.

The solver supports two types of annealing methods ‘annealing’ or ‘parallel_tempering’, specified by the parameter optimization_method.

optimization_method = 'annealing':

The annealing is a random walk of length number_iterations. number_runs independent runs are executed. On CPU all runs are executed sequentially so that number_runs can be set to an arbitrary integer.

The Digital Annealer reduces the temperature during the annealing process. The temperature is kept constant for a number of iterations given as temperature_interval. For temperature_interval=1 the temperature is updated after every step. Digital Annealer supports three different temperature modes of temperature reduction:

  • temperature_mode=0: reduce temperature by factor (1-temperature_decay)

  • temperature_mode=1: reduce temperature by factor (1-temperature_decay*temperature)

  • temperature_mode=2: reduce temperature by factor (1-temperature_decay*temperature^2)

temperature_decay has to be a value between 0 and 1 for temperature_mode=0 and between 0 and 1/temperature_start respectively 1/temperature_start^2 for the other modes. For bigger values the temperature declines quicker. The annealing always starts at temperature temperature_start. If a float value for temperature_end is defined, then a decay is determined automatically to reach this temperature in the given steps. If temperature_end is None then the parameter temperature_decay is used as decay.

optimization_method = 'parallel_tempering':

In parallel tempering mode, number_replica many independent annealings of length number_iterations are executed. On CPU number_replica can be set to an arbitrary integer. The random walks are also referred to as furnaces and have different constant annealing temperatures as specified by the parameter pt_temperature_model. Options are ‘PT_TEMPERATURE_MODEL_LINEAR’ or ‘PT_TEMPERATURE_MODEL_EXPONENTIAL’ to have linear respectively exponential decrease of temperatures between temperature_start and temperature_end along the furnaces, or ‘PT_TEMPERATURE_MODEL_HUKUSHIMA’ to have temperatures in the furnaces which imply a similar replica exchange probability for all neighbouring furnaces. Every qubo bit number iterations, the current replicas (states) are randomly exchanged between the random walks. The exchange happens according to the chosen pt_replica_exchange_model. Possible options are ‘PT_REPLICA_EXCHANGE_MODEL_NEIGHBOURS’ to randomly exchange replicas of neighboring furnaces, or ‘PT_REPLICA_EXCHANGE_MODEL_FAR_JUMP’ to exchange replicas of random pairs of furnaces.

When the random walk in the annealing or parallel tempering process walks into a local minimum it may happen that no candidate is determined for the next step. To rapidly overcome this waiting situation and escape from the local minimum the DA can increase temporarily and stepwise the energy (dynamic energy). This increase is defined by offset_increase_rate. A value of 0.0 switches off the dynamic energy feature.

With guidance_config it is possible to define the bit setting at the beginning of the annealing process.

For the CPU platform data for graphical analysis of the annealing can be produced by setting the parameter graphics=True. The graphics are included in the SolutionList, and can be inspected via the dadk.Solution_SolutionList.SolutionList().

The parameters scaling_action, scaling_factor and scaling_bit_precision are used for scaling the qubo and annealing temperatures.

In order to run the annealing core on multiple threads set the environment variable DADK_CPU_ANNEALING_THREADS to the number of cores you want to use.

import os
os.environ['DADK_CPU_ANNEALING_THREADS'] = '4'
Parameters:

  • optimization_method (OptimizationMethod) – Digital Annealer optimization algorithms, ‘annealing’ or ‘parallel_tempering’. GUI: Optimization method , Default: OptimizationMethod.annealing

  • number_runs (int) – Number of stochastically independent runs. GUI: Number of runs , Default: 2, Min: 1, Max: 128

  • number_replicas (int) – Number of replicas in parallel tempering. GUI: Number of replicas , Default: 5, Min: 5, Max: 128

  • number_iterations (int) – Total number of iterations per run. GUI: Number of iterations , Default: 1000, Min: 1, Max: 100000000

  • temperature_sampling (bool) – Temperatures GUI: Temperatures , Default: True

  • temperature_start (float) – Start temperature of the annealing process. GUI: Temperature start , Default: 1000.0, Min: 0.0, Max: 100000000000000000000.0

  • temperature_end (float) – End temperature of the annealing process. GUI: Temperature end , Default: 1.0, Min: 0.0, Max: 100000000000000000000.0

  • temperature_mode (int) –

    0, 1, or 2 to define the cooling curve:

    0, ‘EXPONENTIAL’:

TemperatureModelHukushima(TemperatureModelExponential)#

class dadk.QUBOSolverBase.TemperatureModelHukushima(temperature_high: float, temperature_low: float, replica_count: int, qubo: ~dadk.BinPol.BinPol, randomwalk_count: int = 10, randomwalk_length: int = 1000, fix_point_precision: float = 1e-05, max_fixpoint_iterations: int = 1000, temperature_adjustment_count: int = 42, logger: ~logging.Logger = <Logger dadk.QUBOSolverBase (ERROR)>)#

dadk.internal.QUBOSolverBase.TemperatureModelHukushima provides a temperature list for parallel tempering furnaces with statistically similar exchange probabilities between neighboring furnaces with their respective temperatures.

A list of temperatures for the furnaces is created. It consists of replica_count elements starting with temperature_high and ending with temperature_low. For two indices i < j the exchange probability is defined by p_ex = min( 1, exp((E_i-E_j) * (1/T_i - 1/T_j)). Starting with an exponential model by TemperatureModelExponential the energies for each furnace are sampled. The temperatures are adjusted iteratively to minimize the difference of exchange probabilities while correcting the energies via interpolation after each step. The count of temperature adjustments is defined by temperature_adjustment_count. The adjustment uses a fixpoint algorithm which can be controlled with the parameters fix_point_precision```and ``max_fixpoint_iterations.

Parameters:

  • temperature_high (float) – Highest temperature.

TemperatureModelLinear(TemperatureModel)#

class dadk.QUBOSolverBase.TemperatureModelLinear(temperature_high: float, temperature_low: float, replica_count: int, logger: ~logging.Logger = <Logger dadk.QUBOSolverBase (ERROR)>)#

Objects of this class provide a temperature list for parallel tempering furnaces with linear decrease.

A list of temperatures for the furnaces is created. It consists of replica_count elements starting with temperature_high and ending with temperature_low. A decay is automatically determined such that every two sub sequential temperatures have the same difference. Both temperatures must be greater than 0, and temperature_low is smaller or equal than temperature_high.

Parameters:

  • temperature_high (float) – Highest temperature.

  • temperature_low (float) – Lowest temperature.

  • replica_count (int) – Number of replicas.

  • logger (Logger) – Logger used for this model.
